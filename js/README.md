# lm_math_fi — JavaScript fixed-point port

A single, zero-dependency ES module that reproduces the fixed-point reference
model in `model/lm_math_fi_model` **bit-for-bit**. The Python model is a thin
wrapper over `fxpmath==0.4.10`; this port reproduces fxpmath's conversion
pipeline exactly, including the points where fxpmath is float64-lossy.

This directory is the shippable artifact for vendoring into a separate website:

| File | Purpose |
|---|---|
| `lm_math_fi.mjs` | The library. No build step, no bundler, no runtime dependencies. Serve it as a static file. |
| `golden_vectors.json` | Golden vectors emitted from the Python/fxpmath model. A deterministic, byte-reproducible oracle (safe to sha256-pin). |
| `test/golden.test.mjs` | Node built-in test runner that replays every golden vector and asserts the library reproduces it. Dev-only. |

## Provenance and determinism

The golden vectors are generated by `scripts/gen_js_golden_vectors.py` from the
Python model. The file is a **pure function of the generator and the installed
fxpmath version** — it contains no git SHA, timestamp, or RNG-derived field — so
re-running the generator reproduces it **byte-for-byte**, and it is safe to pin by
content hash downstream.

- `fxpmath` version: **0.4.10**, recorded in the top-level `fxpmath_version` field
  (the generator asserts the installed version matches before emitting).

Regenerate with `python scripts/gen_js_golden_vectors.py`, and verify the committed
file is byte-identical with `python scripts/gen_js_golden_vectors.py --check` (the
check compares the full bytes; nothing is masked).

## Usage

```js
import { FiFormat, fi, bits, raw, quantize, add, sub, mul, mult_add }
  from "./lm_math_fi.mjs";

// FiFormat(width, binpnt, signed)
const q4_2u = new FiFormat(4, 2, false);

fi(1.5, q4_2u).bits;                  // "0110"
quantize(bits("10111", new FiFormat(5, 2, true)),
         new FiFormat(5, 1, true),
         { rounding: "round" }).bits;  // "11100"

mul(fi(1.5, q4_2u), fi(0.75, q4_2u),
    new FiFormat(8, 4, false)).toFloat();
```

The vocabulary matches the Python model exactly: `width` total bits, `binpnt`
fractional bits, `signed` selects two's-complement. The same rounding keys
(`trunc_bits`, `trunc`, `bit_trunc`, `trunc_zero`, `fix`, `round`, `round_even`,
`nearest_even`, `round_pos_inf`, `round_neg_inf`, `round_zero`, `round_away`,
`round_inf`, `floor`, `ceil`, and the `nearest_*` spellings) and overflow keys
(`wrap`, `saturate`) are accepted. `add`/`sub`/`mul`/`mult_add` and `quantize`
take `{ rounding, overflow }`; `mult_add` also takes `{ subtract }`.

### Accessors

- `bits` — two's-complement bit string of length `width`
- `raw_signed` / `raw_unsigned` — **BigInt** raw values (exact beyond 53 bits)
- `hex` — `0x`-prefixed, zero-padded to `ceil(width/4)` digits, from the unsigned raw
- `toFloat()` — Number (`raw / 2**binpnt`)

### Inputs

- `Number` — taken through the float64 path (same as a Python `float`).
- `BigInt` — exact integer input (same as a Python `int`); use this for exact
  integers above 2**53, which a `Number` cannot represent.
- string — a decimal string (`"1.5"`) goes through the float path; a based
  literal (`"0b1011"`, `"0x1f"`) is parsed like `bits`/`raw`.

## Matching fxpmath, not "fixing" it

The port deliberately reproduces fxpmath's float64 pipeline rather than computing
exact rational arithmetic. fxpmath converts an input by coercing it to a float64,
computing `value * 2**binpnt` in float64 (a power-of-two multiply only shifts the
exponent, so it is exact), applying the rounding mode in float64, then applying
overflow and masking. JS `Number` is the same IEEE-754 double as numpy float64,
and decimal-string-to-double is correctly rounded in both, so results match.

Trying to be "more correct" (exact decimal rounding) would *diverge* from the
reference for non-dyadic inputs such as `0.1`. Bit-for-bit fidelity requires the
float64 approximation first.

## Supported domain / high-width boundary

Reproduced exactly (matching fxpmath, including where fxpmath is lossy):

- **Float / decimal input**: lossless while the scaled raw magnitude stays within
  2**53; above that the result is float64-lossy but deterministic, and the port
  reproduces fxpmath's exact lossy result.
- **Integer input via BigInt**: exact at any magnitude.
- **Bit-string / raw input, `binpnt == 0`**: exact at any width (verified at
  53/60/64/65 bits).
- **Bit-string / raw input, `binpnt > 0`**: fxpmath parses the integer exactly
  then divides by `2**binpnt` in float64, so it is float64-lossy once the raw
  magnitude exceeds 2**53. The port reproduces this (e.g. an all-ones unsigned
  54/4 value collapses to raw 0, matching fxpmath). Use widths ≤ 53 for
  fractional bit/raw input where losslessness is required.
- **Arithmetic intermediates** are held exactly in BigInt and combined using
  fxpmath's growing result format (and its unsigned wraparound). Requantization
  that reduces fractional bits reproduces fxpmath's `float(raw) * 2**delta`
  float64 step; requantization that keeps or grows fractional bits is exact.

Out-of-domain cases (where fxpmath itself raises, e.g. `OverflowError` when
requantizing a value wider than 64 bits down to a sub-64-bit width, or `ValueError`
when rejecting a malformed literal) are **recorded as expected-error oracle entries
and replayed/asserted by CI**: the library must throw, and the thrown error's
category must match the category fxpmath's exception was mapped to (see the schema
below). They are part of the contract, not skipped.

## Golden vector schema

`golden_vectors.json` is self-describing enough to replay without the test runner.

**Top level**

| Field | Meaning |
|---|---|
| `schema` | Format id, currently `"lm_math_fi.golden.v1"`. |
| `fxpmath_version` | The fxpmath version the vectors were generated against (`"0.4.10"`). |
| `rounding_keys` / `overflow_keys` | The accepted rounding / overflow mode names. |
| `rtl_overlap_checked` | `true` if the generator cross-checked the VHDL-overlap vectors. |
| `count` | Number of entries in `vectors` (equals `vectors.length`). |
| `vectors` | Array of vector objects (below). |

**Per vector**

- `id` — 1-based sequence number.
- `op` — one of `fi`, `bits`, `raw`, `quantize`, `add`, `sub`, `mul`, `mult_add`.
- `note` — human-readable label (informational).
- `spec` — the operation's inputs (see shapes below).
- `expect` — either a value result or an error result (see below).

**Formats.** Every format is `fmt = [width, binpnt, signed]` (a 3-element array):
`width` total bits, `binpnt` fractional bits, `signed` a boolean.

**Constructors** (used directly for `fi`/`bits`/`raw`, and nested inside operation
specs as operands):

- `{ "ctor": "fi",   "value": <value>, "fmt": [w,b,s], "rounding": <key>, "overflow": <key> }`
- `{ "ctor": "bits", "value": "<bitstring>", "fmt": [w,b,s] }`
- `{ "ctor": "raw",  "value": "<decimal-int-string>", "fmt": [w,b,s] }`

**Value encoding** (the `value` of a `fi` constructor) is a tagged object:

- `{ "t": "float", "v": <JSON number> }` — pass as a JS `Number` (float64 path).
- `{ "t": "int",   "v": "<decimal string>" }` — pass as a `BigInt` (exact integer).
- `{ "t": "str",   "v": "<string>" }` — pass as a string literal (decimal, or a
  `0b`/`0x`/`0h` based literal).

**Operation `spec` shapes** (the top-level `op` decides which is used):

- `fi` / `bits` / `raw`: `spec` *is* the constructor object above.
- `quantize`: `{ "source": <ctor>, "fmt": [w,b,s], "rounding": <key>, "overflow": <key> }`.
- `add` / `sub` / `mul`: `{ "left": <ctor>, "right": <ctor>, "fmt": [w,b,s], "rounding": <key>, "overflow": <key> }`.
- `mult_add`: as above with `"left"`, `"right"`, `"addend"` constructors plus an
  optional `"subtract": <bool>`.

**`expect` — value result** (all integers are decimal strings to survive >53-bit
values):

```json
{ "bits": "<two's-complement bitstring, length = width>",
  "raw_signed": "<decimal string>",
  "raw_unsigned": "<decimal string>",
  "hex": "0x<UPPER, ceil(width/4) digits>",
  "float": <JSON number> }
```

**`expect` — error result** (fxpmath raised on this input; the library must throw
with a matching category):

```json
{ "error": { "category": "<canonical category>", "py_type": "<Python exception class>" } }
```

`category` is derived from the Python exception type (never assigned per case):
`ValueError → "parse_error"`, `OverflowError → "domain_error"`. The library throws
an `FiError` whose `.category` (and `.code`) carries the same value; `py_type` is
informational provenance.

**Worked example — value vector**

```json
{ "id": 6320, "op": "bits", "note": "ut bits_and_raw_values",
  "spec": { "ctor": "bits", "value": "10111", "fmt": [5, 2, true] },
  "expect": { "bits": "10111", "raw_signed": "-9", "raw_unsigned": "23",
              "hex": "0x17", "float": -2.25 } }
```

Replay: construct from bits `10111` in format `(width=5, binpnt=2, signed=true)`;
the stored value is `-9` raw (`-2.25`).

**Worked example — expected-error vector**

```json
{ "id": 6203, "op": "fi", "note": "review: fi 0b10000 (4,0,u) raises",
  "spec": { "ctor": "fi", "value": { "t": "str", "v": "0b10000" }, "fmt": [4, 0, false],
            "rounding": "trunc_bits", "overflow": "wrap" },
  "expect": { "error": { "category": "parse_error", "py_type": "ValueError" } } }
```

Replay: `fi("0b10000", (4,0,unsigned))` must throw a `parse_error` (the literal is
wider than the 4-bit word).
